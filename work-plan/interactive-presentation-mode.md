# MVM Interactive 演示模式设计计划

## 一、问题分析

### 1.1 当前 Interactive 渲染器的局限

**现状**：
- Interactive 渲染器支持丰富的交互组件（手风琴、标签页、时间线、闪卡、测验、揭示等）
- 每个组件都需要用户主动点击才能交互
- 浏览者需要理解每个组件的交互方式

**问题**：
- 制作者预演一遍后，浏览者无法直接复现相同的体验
- 浏览者需要学习每个组件的交互方式
- 无法像 PPT 那样"一键演示"

### 1.2 用户需求

**核心需求**：
1. **制作者侧**：可以预演一遍交互流程，记录点击顺序
2. **浏览者侧**：只需要一个重复动作（向下翻页/点击前进）就能参与
3. **体验一致性**：制作者看到的 = 浏览者看到的

**类比场景**：
- PPT 的演示模式：制作者按顺序设计，浏览者按顺序观看
- 视频的播放：制作者录制，浏览者播放
- 游戏的通关路线：制作者设计，浏览者按路线游玩

## 二、实现层面选择

### 2.1 三种可能的实现层面

| 层面 | 方案 | 优点 | 缺点 |
|------|------|------|------|
| **MSG 层**（内容层） | 在 .mvm 文件中定义交互序列 | 内容可控，精确控制流程 | 违反"内容与呈现分离"，增加格式复杂度 |
| **View 层**（渲染器层） | 在 InteractiveRenderer 中实现演示模式 | 不改变 .mvm 格式，保持内容纯粹 | 不同渲染器需要各自实现 |
| **扩展层**（附件层） | 创建独立的交互脚本文件 | 灵活，不污染核心格式 | 增加文件管理复杂度 |

### 2.2 推荐选择：View 层实现

**选择理由**：

1. **符合 MVM 核心理念**
   - 内容与呈现分离
   - Model/Msg 纯粹存储内容
   - View 负责所有呈现逻辑，包括交互流程

2. **不增加 .mvm 格式复杂度**
   - .mvm 文件保持纯粹的内容定义
   - 交互流程是"呈现方式"，不是"内容本身"

3. **渲染器职责明确**
   - InteractiveRenderer 本身就是"交互式渲染器"
   - 演示模式是其自然的功能扩展
   - 其他渲染器（Web/MD/Slide）不需要实现

4. **实现简单**
   - 只需在 InteractiveRenderer 中添加演示模式
   - 不需要修改解析器、Schema、其他渲染器

## 三、技术方案设计

### 3.1 核心概念

**演示模式（Presentation Mode）**：
- 制作者在"编辑模式"下预演一遍，记录所有交互
- 系统生成"交互序列"（Interaction Sequence）
- 浏览者在"演示模式"下，按顺序回放交互序列
- 浏览者只需一个动作（前进/后退/播放/暂停）

### 3.2 交互序列数据结构

```javascript
InteractionSequence = [
  {
    sectionId: "section-1",
    action: "open-accordion",
    index: 0,
    timestamp: 1000
  },
  {
    sectionId: "section-2",
    action: "switch-tab",
    index: 1,
    timestamp: 3000
  },
  {
    sectionId: "section-3",
    action: "next-flashcard",
    index: 0,
    timestamp: 5000
  }
]
```

**字段说明**：
- `sectionId`：目标 section 的 ID
- `action`：交互类型（open-accordion, switch-tab, next-flashcard, reveal, etc.）
- `index`：子元素索引（如手风琴的第几项、标签页的第几页）
- `timestamp`：制作者预演时的时间戳（可选，用于自动播放）

### 3.3 两种模式

#### 编辑模式（Edit Mode）

**制作者侧**：
- 可以自由点击所有交互组件
- 系统自动记录每次交互
- 可以重新录制、清空记录
- 可以预览交互序列

**功能按钮**：
- [开始录制] [停止录制] [清空记录] [预览序列]

#### 演示模式（Presentation Mode）

**浏览者侧**：
- 只能看到"前进"按钮（或键盘空格键）
- 点击"前进"自动执行交互序列中的下一步
- 支持"后退"回退到上一步
- 支持"自动播放"按时间戳自动执行

**功能按钮**：
- [前进] [后退] [播放] [暂停] [重置]

### 3.4 交互类型映射

| 组件类型 | 交互动作 | 记录方式 |
|----------|----------|----------|
| **手风琴**（accordion） | 打开/关闭某一项 | 记录 `open-accordion` + `index` |
| **标签页**（tab） | 切换到某一页 | 记录 `switch-tab` + `index` |
| **时间线**（timeline） | 滚动到某一项 | 记录 `scroll-to` + `index` |
| **闪卡**（flashcard） | 翻到下一张 | 记录 `next-flashcard` + `index` |
| **测验**（quiz） | 选择答案 | 记录 `select-answer` + `index` |
| **揭示**（reveal） | 显示隐藏内容 | 记录 `reveal` + `sectionId` |
| **表格**（table） | 排序/筛选 | 记录 `sort` + `column` |
| **列表**（list） | 展开/折叠 | 记录 `toggle-list` + `index` |

## 四、实现计划

### Phase 1：基础架构（1-2天）

**目标**：建立交互序列的记录和回放框架

| 任务 | 优先级 | 说明 |
|------|--------|------|
| 定义 InteractionSequence 数据结构 | 高 | JSON 格式，包含 sectionId, action, index, timestamp |
| 实现 InteractionRecorder 类 | 高 | 负责记录用户的交互动作 |
| 实现 InteractionPlayer 类 | 高 | 负责按顺序回放交互序列 |
| 添加模式切换功能 | 高 | 编辑模式 ↔ 演示模式切换 |
| 添加录制控制按钮 | 中 | 开始/停止/清空/预览 |

**交付物**：
- `InteractionRecorder.js`
- `InteractionPlayer.js`
- 基础的录制/回放功能

### Phase 2：组件集成（2-3天）

**目标**：让所有交互组件支持录制和回放

| 任务 | 优先级 | 说明 |
|------|--------|------|
| 手风琴组件集成 | 高 | 记录打开/关闭动作，支持回放 |
| 标签页组件集成 | 高 | 记录切换动作，支持回放 |
| 时间线组件集成 | 中 | 记录滚动动作，支持回放 |
| 闪卡组件集成 | 中 | 记录翻卡动作，支持回放 |
| 测验组件集成 | 中 | 记录选择动作，支持回放 |
| 揭示组件集成 | 低 | 记录显示动作，支持回放 |

**交付物**：
- 所有组件支持 `onInteraction` 回调
- 所有组件支持 `executeInteraction(action)` 方法

### Phase 3：演示模式 UI（1-2天）

**目标**：提供友好的演示模式界面

| 任务 | 优先级 | 说明 |
|------|--------|------|
| 演示模式控制栏 | 高 | 前进/后退/播放/暂停/重置按钮 |
| 进度指示器 | 中 | 显示当前步骤 / 总步骤 |
| 键盘快捷键 | 中 | 空格前进，左右箭头后退/前进 |
| 自动播放功能 | 低 | 按时间戳自动执行交互 |

**交付物**：
- 演示模式控制栏 UI
- 进度指示器
- 键盘快捷键支持

### Phase 4：序列持久化（1天）

**目标**：保存和加载交互序列

| 任务 | 优先级 | 说明 |
|------|--------|------|
| 序列导出功能 | 高 | 导出为 JSON 文件 |
| 序列导入功能 | 高 | 导入 JSON 文件并回放 |
| 序列嵌入 .mvm 文件 | 中 | 可选：将序列作为注释嵌入 .mvm |

**交付物**：
- 导出/导入功能
- 可选的序列嵌入方案

### Phase 5：优化和测试（1-2天）

**目标**：提升用户体验，确保稳定性

| 任务 | 优先级 | 说明 |
|------|--------|------|
| 动画过渡 | 中 | 交互之间的平滑过渡 |
| 错误处理 | 高 | 组件不存在、索引越界等错误 |
| 性能优化 | 中 | 大型交互序列的性能 |
| 用户测试 | 高 | 真实用户测试，收集反馈 |

**交付物**：
- 优化的演示模式体验
- 完整的错误处理
- 性能优化报告

## 五、数据流设计

### 5.1 录制流程

```
制作者在编辑模式下操作
    ↓
用户点击某个交互组件（如手风琴）
    ↓
组件触发 onInteraction 回调
    ↓
InteractionRecorder 记录交互
    ↓
添加到 InteractionSequence 数组
    ↓
实时更新 UI（显示已记录的交互数）
```

### 5.2 回放流程

```
浏览者在演示模式下点击"前进"
    ↓
InteractionPlayer 读取下一步交互
    ↓
查找对应的 section 和组件
    ↓
调用组件的 executeInteraction(action) 方法
    ↓
组件执行交互（如打开手风琴）
    ↓
更新进度指示器
    ↓
准备下一步交互
```

### 5.3 自动播放流程

```
浏览者点击"播放"
    ↓
InteractionPlayer 按时间戳排序交互
    ↓
计算每个交互的延迟时间
    ↓
使用 setTimeout 按顺序执行
    ↓
支持暂停/继续/停止
```

## 六、边界情况处理

### 6.1 组件不存在

**场景**：交互序列中引用的 section 或组件不存在

**处理**：
- 跳过该交互，继续下一步
- 在控制台输出警告
- 在 UI 上显示"部分交互无法执行"提示

### 6.2 索引越界

**场景**：交互序列中的 index 超出当前组件的范围

**处理**：
- 跳过该交互，继续下一步
- 在控制台输出警告
- 在 UI 上显示"部分交互无法执行"提示

### 6.3 交互冲突

**场景**：当前组件状态与交互序列中的动作冲突

**处理**：
- 强制执行交互序列中的动作
- 忽略当前组件状态
- 确保演示模式的一致性

### 6.4 用户手动干预

**场景**：浏览者在演示模式下手动点击组件

**处理**：
- 禁用手动点击（推荐）
- 或者记录手动点击，但不影响交互序列（备选）

## 七、扩展性考虑

### 7.1 新组件支持

**原则**：
- 所有新组件必须实现 `onInteraction` 回调
- 所有新组件必须实现 `executeInteraction(action)` 方法
- 新的交互类型添加到 InteractionType 枚举

### 7.2 多种演示模式

**未来扩展**：
- **引导模式**：制作者添加文字说明，指导浏览者操作
- **分支模式**：制作者定义多个交互路径，浏览者选择
- **互动模式**：浏览者的选择影响后续内容

### 7.3 跨渲染器支持

**未来扩展**：
- 其他渲染器（Web/Slide）也可以实现演示模式
- 共享 InteractionSequence 数据格式
- 统一的交互录制和回放标准

---

## 八、总结

### 8.1 核心价值

1. **降低浏览门槛**：浏览者只需一个动作（前进）就能参与
2. **提升演示效果**：制作者可以精确控制演示流程
3. **保持内容纯粹**：不污染 .mvm 格式，符合 MVM 理念
4. **易于实现**：只在 InteractiveRenderer 中实现，不影响其他部分

### 8.2 实现路径

```
Phase 1: 基础架构（1-2天）
    ↓
Phase 2: 组件集成（2-3天）
    ↓
Phase 3: 演示模式 UI（1-2天）
    ↓
Phase 4: 序列持久化（1天）
    ↓
Phase 5: 优化和测试（1-2天）
```

**总工期**：6-10天

### 8.3 关键决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 实现层面 | View 层（InteractiveRenderer） | 符合 MVM 理念，不增加格式复杂度 |
| 数据存储 | 内存 + 可选导出 JSON | 灵活，不强制持久化 |
| 用户交互 | 单一动作（前进） | 降低浏览门槛 |
| 自动播放 | 可选功能 | 不强制，按需使用 |

### 8.4 成功标准

- [ ] 制作者可以录制交互序列
- [ ] 浏览者可以通过"前进"按钮回放序列
- [ ] 支持键盘快捷键（空格前进）
- [ ] 支持自动播放
- [ ] 所有交互组件支持录制和回放
- [ ] 边界情况处理完善
- [ ] 用户体验流畅
