# MVM 文件格式设计文档

## 一、MVM 架构定义

MVM = **Model - View - Msg**，一种面向内容呈现的文件架构模式。

### 核心三要素

| 要素 | 职责 | 特性 |
|------|------|------|
| **Model** | 结构定义、类型、规则 | 只存不改，不可变，定义"内容长什么样" |
| **Msg** | 文字、信息、内容 | 纯信息，不含任何样式或布局指令 |
| **View** | 呈现方式、渲染结果 | 只消费 Model+Msg，不修改内容本身 |

### 核心思想

- Model 定义内容的骨架（有哪些字段、什么类型、什么约束）
- Msg 填充内容的血肉（实际文字、数据、信息）
- View 决定内容穿什么衣服（网页、PPT、PDF、动画、语音……）
- **同一份内容，无限种呈现方式，内容本身永远不变**

### 本质：内容层与应用层的分离

Web 本来就该分成两层：

| 层级 | 职责 | 特征 | 代表 |
|------|------|------|------|
| **内容层** | 文章、文档、图表、卡片 | 纯展示，无状态，无逻辑 | MD、Mermaid、SVG、PDF、EPUB |
| **应用层** | 表单、交互、状态、逻辑 | 有状态，有行为，可运行 | Vue、React、Angular |

MVM 属于**内容层**——它只管"内容怎么存、怎么呈现"，不管"用户怎么交互、数据怎么变"。

判断标准很简单：**有没有"可变化的状态"和"可触发的行为"**。

- Vue/React 有：data、computed、watch、emit、onMounted → **真前端（应用层）**
- Mermaid/MD 没有：画完就定了，不会自己变、不会自己跑逻辑 → **伪前端（内容层）**
- MVM 没有：Model 不可变，Msg 纯信息，View 只渲染 → **伪前端（内容层）**

> AI 生成的架构图 = 结构化内容 → 渲染成图，不是前端，是高级版富文本。

## 二、MVM 与内容呈现格式的对比

### 2.1 与 PPT（PowerPoint）的对比

PPT 是目前最主流的内容呈现工具，但它的根本问题在于**内容与样式强绑定**：

| 维度 | PPT (.pptx) | .mvm |
|------|-------------|------|
| 内容与样式 | 混在一起（每页都带样式） | 完全分离（Model+Msg 无样式） |
| 换主题/风格 | 需要逐页修改或重新套模板 | 换一个 View 渲染器即可 |
| 内容复用 | 复制粘贴，容易丢失格式 | 同一 .mvm 文件直接复用 |
| 多设备适配 | 需要手动调整布局 | View 渲染器自动适配 |
| 协作编辑 | 冲突频繁，合并困难 | 纯文本，Git 友好 |
| 版本管理 | 二进制文件，无法 diff | 纯文本，每次改动清晰可见 |
| 动画效果 | 内建，但有限 | 取决于 View 渲染器，无限可能 |
| 输出格式 | 仅 PPT / PDF | 网页、PDF、PPT、动画、语音、卡片…… |

**PPT 的问题本质：它是"所见即所得"的产物，内容写进去的那一刻，呈现方式就被锁死了。**

**MVM 的解法：内容写进去的那一刻，呈现方式还没有被决定——由 View 层在渲染时决定。**

### 2.2 与 Markdown 的对比

Markdown 是文本领域的 MVM 原型，但结构能力有限：

| 维度 | .md | .mvm |
|------|-----|------|
| 结构定义 | 隐式（`# * **` 等符号） | 显式（MODEL 区独立定义） |
| 类型系统 | 无 | 内建（string/text/int/enum/array） |
| 字段约束 | 无 | 有（必填/可选/枚举/嵌套） |
| 多视图渲染 | 有限（主要是文本类） | 无限（文本/动画/语音/3D/游戏） |
| 程序解析 | 模糊，依赖正则 | 精确，Schema 驱动 |
| 内容编辑 | 自然 | 同样自然 |

**MD 源码 = 结构符号(Model) + 文字信息(Msg)**，但结构符号太弱，无法精确描述复杂内容。

### 2.3 与 LaTeX 的对比

LaTeX 是学术界的排版标准，同样追求内容与样式分离：

| 维度 | .tex (LaTeX) | .mvm |
|------|-------------|------|
| 内容与样式 | 分离（.tex + .cls/.sty） | 分离（MODEL + MSG，View 外部） |
| 学习曲线 | 极陡（宏语言、包管理） | 平缓（类 Markdown 语法） |
| 渲染目标 | 主要为 PDF | 任意（PDF/网页/PPT/动画/语音） |
| 实时预览 | 需要编译 | 可流式渲染 |
| 错误处理 | 编译报晦涩错误 | Schema 验证，清晰报错 |
| 扩展性 | 通过宏包扩展 | 通过 View 渲染器扩展 |

### 2.4 与 HTML 的对比

HTML 是网页的呈现标准，但内容与样式严重耦合：

| 维度 | .html | .mvm |
|------|-------|------|
| 内容与样式 | 耦合（虽然 CSS 分离了样式，但 HTML 标签本身携带布局语义） | 完全分离（MSG 区纯内容，无任何标签） |
| 语义化 | 通过标签实现（`<h1>` `<p>` `<div>`） | 通过 Model 类型系统实现 |
| 多视图 | 仅网页 | 任意 |
| 编辑体验 | 标签噪音大 | 接近纯文本写作 |

### 2.5 与前端框架（Vue/React）的对比

Vue/React 是应用层框架，MVM 是内容层格式，两者不在同一层级：

| 维度 | Vue / React | .mvm |
|------|-------------|------|
| 定位 | 应用层（状态+逻辑+渲染） | 内容层（结构+数据+呈现） |
| 状态管理 | 有（data/store/state） | 无（Model 不可变） |
| 事件交互 | 有（emit/click/onChange） | 无（纯展示） |
| 生命周期 | 有（mounted/updated/unmounted） | 无（一次性渲染） |
| 适用场景 | 交互式应用（后台、表单、SPA） | 内容呈现（文档、幻灯片、卡片） |
| 关系 | 可以作为 MVM 的 View 渲染器 | 可以输出给 Vue/React 组件消费 |

> Vue/React 的终极形态 `UI = f(state)`，View 只是 State 的函数，不存状态、不存逻辑——这正是 MVM 的思想：Model + Msg → View 纯渲染。

### 2.6 综合对比总览

| 维度 | PPT | MD | LaTeX | HTML | Vue/React | .mvm |
|------|-----|-----|-------|------|-----------|------|
| 层级 | 内容层 | 内容层 | 内容层 | 内容层 | 应用层 | 内容层 |
| 内容与样式分离 | 差 | 中 | 良 | 中 | 中 | 优 |
| 多视图支持 | 差 | 中 | 差 | 差 | 中 | 优 |
| 编辑体验 | 中 | 优 | 差 | 差 | 中 | 优 |
| 结构化程度 | 中 | 弱 | 强 | 强 | 强 | 强 |
| 学习成本 | 低 | 低 | 高 | 中 | 高 | 低 |
| 版本管理 | 差 | 优 | 优 | 优 | 中 | 优 |
| 动画/交互 | 中 | 无 | 无 | 强 | 强 | 取决于 View |
| 内容复用 | 差 | 中 | 中 | 中 | 中 | 优 |

## 三、MVM 的行业定位与可行性分析

### 3.1 为什么"内容层独立"没有成为主流？

历史上不是没人尝试过内容层独立，而是被商业和生态锁死了：

| 历史方案 | 结果 | 原因 |
|----------|------|------|
| **XAML** (WPF/MAUI) | 被边缘化 | 太重、太复杂，被 Web 生态碾压 |
| **QML** (Qt Quick) | 小众 | 绑定 Qt 生态，没有独立标准 |
| **HTML 最初设计** | 被异化 | 从内容标记变成了应用平台 |
| **Headless UI** | 正在兴起 | 还没有统一的内容格式标准 |

根本原因：

1. **框架公司靠捆绑赚钱**：Vue/React/Angular 把状态+模板+运行时+生态+工具链打包卖，一旦把 View 纯渲染拆出来独立成标准，框架就不再不可替代
2. **大公司都在做应用**：Google/Microsoft/Apple/Meta 都在做应用层，不做内容呈现标准
3. **行业教育偏差**：长期教育大家"UI 必须和逻辑绑在一起"，很少有人敢公开说大部分界面根本不需要状态，只需要渲染内容

### 3.2 MVM 的差异化优势

相比历史上所有方案，MVM 的独特定位：

| 对比 | XAML/QML | HTML | MD | MVM |
|------|----------|------|-----|-----|
| 重量 | 重 | 中 | 轻 | 轻 |
| 结构 | 强但复杂 | 强但混乱 | 弱 | 强且简单 |
| 框架绑定 | 绑定平台 | 绑定浏览器 | 无 | 无 |
| 渲染目标 | 特定平台 | 仅网页 | 文本变体 | 任意 |
| 编辑体验 | 差（XML） | 差（标签） | 优 | 优 |

### 3.3 MVM 的真实价值场景

MVM 不是要取代 PPT 或 Markdown，而是填补一个**被忽视的空白**：

1. **内容创作者**：写一次，自动生成网页+PPT+语音+卡片，不再为每种平台重新排版
2. **技术团队**：用结构化格式写文档，程序可以精确解析和渲染，支持自动化发布
3. **AI 时代**：AI 生成结构化内容（Model+Msg），View 渲染器负责呈现，内容与呈现完全解耦
4. **多平台分发**：同一份内容自动适配公众号、知乎、掘金、PPT、PDF 等多种输出

### 3.4 风险与局限

| 风险 | 说明 | 应对 |
|------|------|------|
| 冷启动问题 | 没有生态和用户基础 | 从特定场景切入（如技术文档自动多端发布） |
| 标准竞争 | MD/LaTeX 已有大量用户 | 不取代，做补充——MD 的结构化升级版 |
| 渲染器工作量 | 每种 View 都需要单独实现 | 先做核心渲染器（Web/MD/Slide），社区扩展 |
| 认知成本 | 新格式需要教育市场 | 语法尽量接近 MD，降低学习门槛 |

## 四、MVM 文件格式设计（.mvm）

### 4.1 设计动机

现有内容呈现格式的根本困境：

- **PPT**：内容与样式强绑定，换风格等于重做
- **Markdown**：结构太弱，无法驱动精确的多视图渲染
- **LaTeX**：学习成本高，渲染目标单一（PDF）
- **HTML**：标签噪音大，本质是网页专用格式

**需要一种新格式：内容写一次，呈现方式无限切换 → .mvm 文件**

### 4.2 .mvm 文件结构

.mvm 文件由两个核心区块组成：

```
===MODEL===
（结构定义区：类型、字段、规则、元信息）
（告诉程序：这份内容有哪些部分、每部分是什么类型）

===MSG===
（信息内容区：纯文字、数据、内容）
（给人看和给渲染器看的实际内容）
```

### 4.3 完整语法规范

#### 区块分隔符

- `===MODEL===` 开始模型定义区
- `===MSG===` 开始信息内容区
- 文件有且仅有这两个顶级区块
- 区块顺序固定：先 MODEL 后 MSG

#### MODEL 区语法

```mvm
===MODEL===
@type Document
@version 1.0

title: string
author: string
created: datetime
tags: array<string>
sections: array<Section>

@type Section
heading: string
body: text
type: enum(normal|quote|code|warning)
priority: int?
metadata: map<string, string>?
```

语法规则：
- `@type TypeName` 定义类型
- `@version x.y` 定义格式版本
- `field: type` 定义字段
- `type?` 表示可选字段
- `array<T>` 表示数组类型
- `map<K, V>` 表示键值对映射
- `enum(a|b|c)` 表示枚举类型
- 基础类型：`string`, `text`, `int`, `float`, `bool`, `datetime`, `url`
- `text` 与 `string` 的区别：`text` 支持多行长文本，`string` 为单行

#### MSG 区语法

```mvm
===MSG===
title: MVM 架构思想
author: 张三
created: 2026-03-31T00:00:00Z
tags: [架构, 前端, 设计模式]

sections:
  ## heading: 思想概述
  type: normal
  body: |
    Model 存结构，Msg 存信息，View 自由渲染。
    这是 MVM 的核心理念。

  ## heading: 示例
  type: code
  body: |
    一份结构，任意视图。
    同一份数据可以渲染成网页、动画、游戏UI。
```

语法规则：
- `field: value` 赋值，与 MODEL 区的字段一一对应
- `|` 表示多行文本块（缩进对齐）
- `[a, b, c]` 表示数组
- `##` 引用已定义的类型（Section），表示嵌套对象
- 注释以 `//` 开头
- 空行用于分隔同级元素

## 五、MVM 的呈现方式（View）

同一份 .mvm 文件可以渲染为以下任意形式，内容本身不需要任何修改：

### 5.1 文档类呈现

| 呈现方式 | 说明 | 对应现有工具 |
|----------|------|-------------|
| 网页 | HTML + CSS，响应式布局 | WordPress / Hugo / Notion |
| PDF | 高质量排版输出 | LaTeX / Word 导出 |
| PPT 幻灯片 | 自动分页，支持动画过渡 | PowerPoint / Keynote |
| 电子书 | EPUB 格式，支持目录和翻页 | Kindle / Apple Books |
| 公众号文章 | 微信排版，自动适配 | 秀米 / 135编辑器 |
| 技术博客 | 带代码高亮和目录 | 知乎 / 掘金 / 博客园 |

### 5.2 动态类呈现

| 呈现方式 | 说明 | 特点 |
|----------|------|------|
| 打字机效果 | 文字逐字出现 | 无需点击，自动播放 |
| 跑马灯 | 文字滚动循环 | 适合公告/通知 |
| 渐显动画 | 内容分步淡入 | 适合演讲/教学 |
| 流式输出 | 模拟 AI 回答效果 | 逐句出现，带光标 |
| 自动轮播 | 多个 section 自动切换 | 适合展示/展览 |

### 5.3 语音/多模态呈现

| 呈现方式 | 说明 | 技术 |
|----------|------|------|
| 语音播报 | TTS 朗读全文 | Edge TTS / Azure |
| 播客格式 | 带开场白的音频 | 文字转语音 + 音频编辑 |
| 视频字幕 | 自动生成字幕文件 | SRT / VTT 格式 |
| 知识卡片 | 每个段落一张卡片 | 小程序 / Instagram |

### 5.4 与 PPT 呈现对比的具体场景

**场景：一份技术分享，需要同时产出网页版、PPT版、语音版**

| 步骤 | PPT 方案 | MVM 方案 |
|------|---------|---------|
| 1. 写内容 | 在 PPT 里写，边写边排版 | 在 .mvm 文件里写纯内容 |
| 2. 做网页 | 重新做一份网页 | WebRenderer 自动渲染 |
| 3. 做 PPT | 就是 PPT 本身 | SlideRenderer 自动渲染 |
| 4. 录语音 | 手动念稿 | VoiceRenderer 自动朗读 |
| 5. 改内容 | 三个地方分别改 | 改 .mvm 文件，全部自动更新 |

## 六、与前沿技术的同构关系

### 6.1 内容呈现技术

| 前沿技术 | MVM 对应 | 同构点 |
|----------|---------|--------|
| **Notion / 飞书文档** | Model=Block 结构, Msg=内容, View=网页/卡片/数据库 | 同一内容块，多种呈现视图 |
| **Reveal.js / Slidev** | Model=Markdown, Msg=文字, View=幻灯片 | MD 写内容，自动变 PPT |
| **MDX** | Model=组件类型, Msg=MD内容, View=React组件 | 内容与组件分离 |
| **Astro Islands** | Model=内容框架, Msg=MD内容, View=任意框架组件 | 内容优先，交互按需加载 |
| **Gamma.app** | Model=AI 理解的结构, Msg=用户输入, View=网页/PPT | AI 自动将内容转为多种呈现 |
| **Tome.app** | Model=故事结构, Msg=叙述内容, View=网页/演示 | 叙事内容自动排版 |

### 6.2 前端架构思想

| 前沿技术 | MVM 对应 | 同构点 |
|----------|---------|--------|
| **声明式 UI** (Compose/SwiftUI/React) | Model=State, Msg=Intent, View=纯渲染 | 状态与视图完全分离 |
| **Headless UI** | Model=组件逻辑, Msg=数据, View=任意样式实现 | 逻辑与样式完全分离 |
| **UI = f(state)** | Model+Msg = state, View = f | View 只是数据的纯函数 |
| **跨平台架构** (KMP/Flutter) | Model=共享逻辑, Msg=平台事件, View=原生渲染 | 一份 Model，N 种 View |

### 6.3 历史方案对比

| 历史方案 | MVM 改进 |
|----------|---------|
| **XAML** (WPF) | 去掉 XML 冗余，用类 MD 语法；去掉平台绑定 |
| **QML** (Qt) | 去掉 JS 逻辑绑定，纯内容层；去掉 Qt 依赖 |
| **HTML** | 去掉标签噪音，纯内容；去掉"仅网页"限制 |
| **JSON Schema** | 加上自然语言内容支持；加上多视图渲染能力 |

## 七、解析器与渲染器架构

### 7.1 解析流程

```
.mvm 文件
    ↓
分块器（Blocker）
    ↓ 分离 ===MODEL=== 和 ===MSG===
    ↓
模型解析器（ModelParser）
    ↓ 解析类型定义、字段、约束
    ↓ 生成 Schema 对象
    ↓
内容解析器（MsgParser）
    ↓ 按 Schema 验证内容
    ↓ 生成 DataObject
    ↓
{ schema, data }
    ↓
视图渲染器（ViewRenderer）
    ↓ 接收 schema + data
    ↓ 输出任意呈现格式
```

### 7.2 渲染器接口

每个渲染器实现同一接口：接收 `{ schema, data }`，输出特定格式。

```
WebRenderer    → HTML + CSS（网页呈现）
MdRenderer     → Markdown（文本呈现）
SlideRenderer  → Reveal.js HTML（幻灯片呈现）
PdfRenderer    → PDF（文档呈现）
VoiceRenderer  → TTS 语音合成（语音呈现）
CardRenderer   → 小程序卡片（移动端呈现）
StreamRenderer → 流式打字机效果（动态呈现）
```

## 八、总结

### MVM 的核心价值

1. **写一次，呈现无限**：同一份 .mvm 文件可以渲染成网页、PPT、PDF、语音、动画等任意形式
2. **内容与样式彻底分离**：不像 PPT 那样内容写进去就被样式锁死
3. **人机双友好**：编辑体验接近 Markdown，程序可以精确解析
4. **版本管理友好**：纯文本格式，Git diff 清晰可见
5. **扩展性强**：新增呈现方式只需写新的 View 渲染器
6. **不依赖任何框架**：纯 Python 解析，纯 HTML/CSS 渲染，零前端框架依赖

### MVM 的本质定位

> MVM 不是"另一个前端框架"，而是"内容层的独立标准"。
> 它和 Vue/React 不竞争——Vue/React 可以作为 MVM 的 View 渲染器。
> 它和 Markdown 不竞争——它是 Markdown 的结构化升级。
> 它和 PPT 不竞争——它让 PPT 变成众多输出格式之一。

### 一句话总结

> .mvm 文件 = 结构化的骨架（Model）+ 自然语言的血肉（Msg），呈现方式（View）是无限的外衣。
